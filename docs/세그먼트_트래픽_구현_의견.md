# 사용자 세그먼트·일일 트래픽 — 개발·마케팅 관점 의견

## 요청하신 것 정리

1. **세분화해서 DB로 쌓고 관리 탭에서 보기**
   - 자주 들락날락 거리는 사람 (제품란·사이트 방문 빈도 높음)
   - 장바구니 넣어놓은 사람 (이미 구현됨: 장바구니 이탈 명단)
   - **그중** 장바구니 넣어두고 자주 들락날락 거리는 사람
   - 텔레그램 봇 자주 쓰는 사람
2. **일일 트래픽** 대시보드에서 통계로 보기
3. **텔레그램 봇 트래픽**도 같이 볼 수 있는지

---

## 1. 개발(IT) 관점

### 난이도: 중~중상. 한 번에 다 하면 복잡, 단계 나누면 가능

| 항목 | 난이도 | 필요한 것 |
|------|--------|-----------|
| 장바구니 넣어놓은 사람 | ✅ 이미 구현 | `cart_snapshots` + 관리자 탭 |
| 자주 들락날락 (방문 빈도) | 중 | **방문/세션 로그**가 있어야 함. 지금은 `activity_logs`(결제 단계 위주), `product_views`(상품별 조회수만, user_id·시각 없음)만 있어서 **유저별·일별** “몇 번 왔는지”를 못 셈. → **site_visits** 같은 테이블(유저·방문 시각) 추가 필요. |
| 체류시간 높은 사람 | 중상 | 들어온 시각 + 나간 시각 필요. **beforeunload** 또는 **visibilitychange**로 “나감” 시점 보내거나, 30초마다 “아직 있음” **heartbeat** 보내는 방식. 테이블은 `session_durations`(user_id, started_at, ended_at) 등. |
| 장바구니 + 자주 들락날락 | 중 | 위 두 개가 있으면 **조회로 합치기**만 하면 됨. (cart_snapshots 있는 user_id 중, site_visits 많은 사람) |
| 텔레그램 봇 자주 쓰는 사람 | **봇 연동 필수** | 웹사이트만으로는 불가. **봇 서버가** 유저가 메시지 보낼 때마다 우리 DB에 적어줘야 함. 예: `telegram_events`(user_id 또는 telegram_id, action, created_at). 봇이 Supabase에 insert하거나 우리 API 호출하도록 하면, 그때부터 “봇 트래픽·활성 유저” 대시보드 가능. |

### 추천 단계

- **1단계(지금 바로)**  
  - **일일 트래픽(사이트)**  
    - 지금 있는 `activity_logs`로 **일별 활성 유저 수** (또는 이벤트 수) 집계해서 대시보드에 표시.  
    - 로그인 유저만 포함되지만, “매일 얼마나 웹에서 움직임이 있었는지” 지표로는 충분히 쓸 수 있음.
- **2단계**  
  - **방문 로그 보강**  
    - 메인/쇼핑/장바구니 등 **페이지 진입 시** `site_visits`(user_id, visited_at) 한 줄 insert.  
    - 그러면 “일일 방문자 수”, “유저별 방문 횟수” 가능 → **자주 들락날락** 세그먼트 가능.
- **3단계**  
  - **체류시간**  
    - 페이지 진입 시 시작 시각 저장, 나갈 때 또는 주기 heartbeat로 종료 시각 저장.  
    - “체류시간 높은 사람” 세그먼트 가능.
- **4단계**  
  - **세그먼트 테이블/관리 탭**  
    - “자주 방문”, “장바구니 이탈”, “둘 다”, “텔레그램 활성” 등을 **정의**한 뒤,  
    - 주기적으로(또는 관리 탭 열 때) 집계해서 `user_segments` 같은 테이블에 넣거나,  
    - 관리 탭에서 **필터/탭**으로 “장바구니 있는 사람 중 방문 많은 순” 등 조회.

### 텔레그램 봇 트래픽

- **웹만으로는 불가.**  
- 봇이 돌아가는 서버(또는 서비스)에서 **유저 행동 이벤트**를 우리 DB/API로 보내줘야 함.  
- 그렇게만 되면 “일일 봇 활성 유저 수”, “봇 메시지 수” 같은 지표를 같은 대시보드에 넣는 건 **구현 난이도는 낮음**.

## 2. 마케팅 관점

### 세그먼트별 가치·활용

| 세그먼트 | 마케팅 의미 | 활용 예 |
|----------|-------------|----------|
| **자주 들락날락** | 관심은 있지만 결제 전 단계 | 리타겟팅 배너, “오늘만 할인” 등으로 전환 유도 |
| **장바구니만 넣고 이탈** | 구매 의도 높음 | 이미 구현된 명단으로 텔레그램/이메일 “장바구니 할인” 메시지 |
| **장바구니 + 자주 방문** | 전환 가능성 가장 높음 | 1:1 쿠폰, 텔레그램으로 개인화 메시지 |
| **텔레그램 봇 자주 사용** | 채널 접근이 쉬운 유저 | 봇으로 선할인·이벤트 알림, 재방문 유도 |

### 우선순위 제안

1. **일일 트래픽(사이트)**  
   - “어제/오늘 몇 명이 웹에서 움직였는지” 바로 보이게 → 운영·마케팅 공통 KPI.
2. **“장바구니 + 자주 방문”**  
   - 데이터만 쌓으면 조회로 뽑을 수 있어서, 비용 대비 효과 큼.
3. **체류시간**  
   - 구현 비용이 제일 크므로, 방문 빈도·장바구니 세그먼트가 안정된 뒤 추가해도 됨.
4. **텔레그램 트래픽**  
   - 봇 측에서 로그만 넘겨주면, 대시보드에 “봇 일일 활성” 등 하나의 블록으로 넣는 건 간단함.

## 3. 정리

- **가능 여부:**  
  - 사이트 쪽: 방문/세션(및 선택적으로 체류시간) 로그를 조금씩 추가하면, 요청하신 세그먼트는 **전부 DB로 쌓고 관리 탭에서 보는 것까지 가능**합니다.  
  - 텔레그램: **봇이 우리 DB/API에 이벤트를 넣어주는지**가 전제입니다.
- **복잡도:**  
  - 한 번에 “모든 세그먼트 + 체류시간 + 봇” 하면 복잡해 보일 수 있고,  
  - **일일 트래픽 → 방문 로그 → 세그먼트 조회/탭 → 체류시간 → 봇 연동** 순으로 나누면 단계별로 납득 가능한 수준입니다.
- **우선 적용 제안:**  
  - **지금:** 대시보드에 **일일 트래픽(사이트)** 추가 (아래처럼 `activity_logs` 기반으로 구현 가능).  
  - **다음:** 방문 로그 테이블 추가 후 “자주 들락날락” / “장바구니 + 자주 방문” 세그먼트와 관리 탭.  
  - **그 다음:** 텔레그램 봇 측에서 이벤트 전달 방식만 정해주면, 봇 트래픽도 같은 대시보드에 블록 하나로 넣을 수 있습니다.
