# 리뷰·주문 실제 데이터 전환 후 — 당신이 할 일 정리

> **이 문서는:**  
> "리뷰/주문 목업 없애고 실제 DB 쓰자"라고 한 뒤, AI가 한 작업과 **당신이 해야 할 일**, **당신이 결정해서 답해 주면 되는 것**을 초등학생도 이해할 수 있게 정리한 것입니다.

---

## 1. AI가 한 일 (요약)

- **주문·리뷰**  
  - 주문 내역·리뷰 목업 제거 → 실제 Supabase `orders`, `product_reviews` 테이블에서만 데이터를 가져와서 보여줌.
- **결제 흐름**  
  - 결제 시 `orders`에 주문 저장, 배송 정보(이름·전화·주소) 스냅샷 저장, 포인트 사용·장바구니 비우기·완료 페이지 이동.
- **주문/배송 정보**  
  - `orders` 테이블에 `receiver_name`, `receiver_phone`, `shipping_address` 등 배송 스냅샷 컬럼 추가용 SQL 작성.
- **수령인 정보 수정**  
  - 관리자: 주문 탭에서 해당 주문의 수령인·전화·주소 수정 가능.  
  - 고객: 본인 주문 내역(배송 전 주문만)에서 «Изменить данные доставки»로 직접 수정 가능.
- **CS 방어·결제 증거**  
  - 결제 단계에서 "가격 본 시각", "결제 버튼 누른 시각"을 `activity_logs`에 기록.  
  - 주문서에 결제 시점 금액·품목·IP·브라우저 정보 저장.  
  - 결제 실패/에러는 `payment_logs`에 기록.  
  - 관리자 화면에 «활동 로그» 탭 추가 → 타임라인 확인 가능.
- **PG(결제사) 연동 대비**  
  - "주문 만들기"와 "결제 승인" 로직 분리 (`paymentGateway.ts`).  
  - `orders`에 `external_transaction_id`, `raw_response` 컬럼 추가용 SQL 작성.  
  - 주문 상태를 `pending` → `completed` / `failed` / `canceled` 등 PG 표준에 맞게 통일.
- **데이터 안전**  
  - 주문 금액·품목은 **스냅샷**으로 저장 → 관리자가 나중에 상품 가격을 바꿔도 예전 주문 금액은 안 바뀜.  
  - 결제 성공했는데 DB 업데이트만 실패한 경우 로그 남기고, "주문 내역 확인·문의" 안내.  
  - 이름·주소·전화·INN·여권 정보가 다 채워져 있을 때만 주문 저장되도록 필수값 검사 추가.

## 2. 당신이 **꼭 해야 하는 일** (조치)

### 2-1. Supabase에서 SQL 실행하기 (DB에 칸 추가)

코드에서는 이미 "새 칸"에 값을 넣거나 읽도록 되어 있는데, **DB에 그 칸이 없으면** 오류가 납니다.  
그래서 아래 SQL 파일들을 **Supabase 대시보드 → SQL Editor**에서 **한 번씩 실행**해 주어야 합니다.

- **파일 1: `docs/SUPABASE_ORDERS_SHIPPING_COLUMNS.sql`**  
  - **하는 일:** `orders` 테이블에 배송 정보 칸 추가 (주소, 수령인 이름, 전화번호).  
  - **추가:** 주문 시점 개인정보 스냅샷용 `inn`, `passport_series`, `passport_number` 칸도 포함 (이미 실행했어도 같은 파일 다시 실행하면 새 칸만 추가됨).  
  - **어디서:** Supabase 대시보드 들어가서 왼쪽 **SQL Editor** 클릭 → 새 쿼리 열기 → 이 파일 내용 전체 복사해서 붙여넣기 → **Run** 버튼.

- **파일 1.5: `docs/SUPABASE_ORDERS_ADMIN_RLS.sql`**  
  - **하는 일:** 관리자가 **다른 사람 주문**의 수령인 정보를 수정할 수 있게 RLS 정책 추가.  
  - 이걸 실행하지 않으면 관리자 화면에서 «저장»을 눌러도 **RLS 때문에 반영되지 않고**, 화면만 바뀐 것처럼 보일 수 있음.  
  - SQL Editor에서 이 파일 내용 실행 후, 관리자에서 다시 수정·저장 테스트.  
  - **코드:** 관리자 저장 실패 시 서버 에러 메시지를 모달 안에 표시하도록 수정해 두었음.

- **파일 1.6: `docs/SUPABASE_REVIEWS_DELETE_RLS.sql`**  
  - **하는 일:** 리뷰 삭제를 **작성자 본인** 또는 **관리자**만 할 수 있도록 RLS 정책 추가 (`product_reviews`, `review_photos`).  
  - 이걸 실행하지 않으면 상품 상세 페이지에서 «Удалить» 버튼을 눌러도 DB에서 삭제가 거부될 수 있음.

- **파일 1.7: `docs/SUPABASE_CART_SNAPSHOTS.sql`**  
  - **하는 일:** 장바구니 이탈 명단용 테이블 `cart_snapshots` 생성. 로그인 사용자가 장바구니 페이지에 품목을 둔 상태로 두면 저장되고, 주문 완료 시 삭제됩니다. 관리자 «장바구니 이탈» 탭에서 명단 확인·CRM(텔레그램 알림·추가 할인 등) 활용 가능.  
  - SQL Editor에서 실행.

- **파일 2: `docs/SUPABASE_CS_DEFENSE_LOGS.sql`**  
  - **하는 일:**  
    - `activity_logs` 테이블 만들기 (가격 본 시각, 결제 버튼 누른 시각 등 로그).  
    - `payment_logs` 테이블 만들기 (결제 실패/에러 기록).  
    - `orders` 테이블에 새 칸 추가:  
      스냅샷 금액·품목, 클라이언트 IP, 브라우저 정보, 결제사 거래 번호·원문 응답 등.  
  - **어디서:** 같은 SQL Editor에서 이 파일 내용 전체 복사 → 붙여넣기 → **Run**.

**정리:**  
- "SQL 실행" = Supabase 웹사이트 들어가서 SQL Editor 열고, 위 두 파일 내용을 각각 붙여넣고 Run 누르기.  
- 한 번만 하면 되고, 이미 같은 칸이 있으면 "이미 있어요" 같은 메시지가 나올 수 있지만, 보통 에러 없이 넘어갑니다.

### 2-2. 관리자로 쓰는 계정 정하기

- **하는 일:**  
  관리자 페이지(주문 목록, 주문 수령인 수정, 활동 로그 등)를 쓰려면, **그 로그인 계정이 "관리자"로 지정**되어 있어야 합니다.  
  이건 **Supabase DB의 `profiles` 테이블**에서 "이 사람은 관리자다"라고 표시해 주는 겁니다.

- **어떻게:**  
  1. Supabase 대시보드 → **Table Editor** → `profiles` 테이블 선택.  
  2. 관리자로 쓸 계정의 **행(row)** 찾기 (예: 본인 이메일로 가입한 계정).  
  3. 그 행에서 `is_admin` 칸을 **true**로 바꾸고 저장.

**정리:**  
- "내가 쓸 계정" 하나 정해서, Supabase `profiles` 테이블에서 그 계정의 `is_admin`을 true로 바꾸면 됩니다.

### 2-3. (선택) 예전 주문 상태 단어 통일

- **상황:**  
  주문 상태를 "paid" → "completed", "cancelled" → "canceled" 같은 식으로 바꿨습니다.  
  화면에서는 예전 데이터도 새 단어로 읽어서 보여주도록 되어 있어서, **안 해도 동작은 합니다.**

- **하고 싶다면:**  
  Supabase SQL Editor에서 아래처럼 한 번 실행해 보세요.  
  (이미 새 단어로 되어 있으면 실행해도 큰 문제 없고, "0 rows"처럼 나올 수 있습니다.)

```sql
UPDATE public.orders SET status = 'completed' WHERE status = 'paid';
UPDATE public.orders SET status = 'canceled' WHERE status = 'cancelled';
```

**정리:**  
- 꼭 할 필요는 없고, "예전 데이터도 DB에서부터 completed/canceled로 통일하고 싶다"할 때만 하면 됩니다.

### 2-4. 결제·주문 흐름 점검 결과 (참고)

아래 흐름을 코드 기준으로 점검해 두었습니다.

| 구간 | 점검 내용 | 비고 |
|------|-----------|------|
| **주문 생성(체크아웃)** | 필수값(이름·주소·전화·INN·여권) 검사 후 insert. full → minimal → minimalNoTest → absoluteMinimal 4단계 폴백. | 스키마가 부족해도 한 단계는 통과하도록 처리됨. |
| **결제 승인** | `paymentGateway.ts`의 `executePayment()` 호출. 현재는 스텁(즉시 성공). | PG 연동 시 이 함수만 교체. |
| **결제 실패 시** | 주문 상태 `failed`로 update, `payment_logs` 기록, 사용자 알림 후 체크아웃 화면 유지. | — |
| **결제 성공 시** | 주문 상태 `completed` + (선택) external_transaction_id, raw_response. update 실패 시 status만 다시 시도, 그래도 실패하면 payment_logs 남기고 안내 문구 표시. | 완료 페이지로 이동은 그대로 함. |
| **완료 페이지** | `orderId`가 state로 전달될 때만 장바구니 비움. 토스트·배송/텔레그램 안내 문구 표시. | — |
| **고객 주문 내역** | 본인 주문만 조회. 배송 전 주문만 수령인 수정 가능. update 시 `.eq('user_id', userId)`로 본인만. | RLS "본인 주문만"과 일치. |
| **관리자 주문** | 목록 조회·수령인 수정 시 **RLS 정책 필요**. `SUPABASE_ORDERS_ADMIN_RLS.sql` 미실행 시 update가 DB에 반영되지 않음. | 저장 실패 시 모달에 에러 메시지 표시하도록 수정됨. |

**정리:**  
- **관리자에서 수령인 저장이 안 됐다** → `docs/SUPABASE_ORDERS_ADMIN_RLS.sql`을 Supabase SQL Editor에서 실행한 뒤 다시 시도.  
- 실행 후에도 실패하면 모달에 표시되는 에러 문구를 확인하면 원인 파악에 도움이 됩니다.

## 3. 당신이 **결정해서 답해 주면 되는 것**

코드는 이미 다 반영되어 있고, 아래는 "설정이나 정책을 어떻게 할지" 당신이 정하면 되는 부분입니다.

### 3-1. 주소 자동완성 (Dadata)

- **지금 상태:**  
  Dadata 키(`VITE_DADATA_API_KEY`)가 이미 `.env`에 들어 있고, 주소 자동완성을 지금 쓰고 계시면 **추가로 할 일 없습니다.**  
  그대로 두고 쓰시면 됩니다.

- **나중에 키를 바꾸거나 끄고 싶을 때만:**  
  - 키 교체 → `.env`에서 `VITE_DADATA_API_KEY` 값만 새 키로 바꾸기.  
  - 자동완성 끄기 → 키를 비우면 주소는 수동 입력만 됨.

**정리:**  
- 이미 반영되어 있고 잘 쓰고 있으면 **할 일 없음.** (문서에서 "결정해서 답해 주면 되는 것" 목록에 넣은 이유는, 다른 프로젝트/환경에서는 키가 없을 수 있어서였습니다.)

---

### 3-2. PG(결제사) 연동 시점

- **지금 상태:**  
  결제 버튼 누르면 "실제 카드 결제"는 하지 않고, 주문만 DB에 저장되고 결제 완료처럼 흐름만 진행됩니다.  
  나중에 Stripe나 러시아 로컬 PG를 붙일 수 있게, **결제 승인만 담당하는 파일** (`src/lib/paymentGateway.ts`)을 따로 두었습니다.

- **당신이 할 일:**  
  - **지금은 PG 안 붙인다** → 아무것도 안 해도 됨.  
  - **나중에 PG 붙인다** → 그때 `src/lib/paymentGateway.ts` 안의 `executePayment()` 함수만 결제사 API 호출하도록 바꾸면 됩니다.  
  - PG에서 받는 "거래 번호", "전체 응답"은 이미 `orders`의 `external_transaction_id`, `raw_response` 칸에 넣도록 준비해 두었습니다.

**정리:**  
- "언제 실제 카드 결제 붙일지"만 정하면 되고, 붙일 때는 `paymentGateway.ts` 한 군데만 수정하면 됨.

---

### 3-3. 필수 입력값 (INN, 여권)

- **지금 상태:**  
  이름(ФИО), 주소, 전화번호, **INN**, **여권(시리즈+번호)** 가 다 채워져 있어야만 "다음" / "주문 확정" 버튼이 활성화되고, 주문이 DB에 저장됩니다.

- **당신이 할 일:**  
  - **INN·여권까지 꼭 받고 싶다** → 지금처럼 두면 됨.  
  - **INN·여권은 선택으로 바꾸고 싶다** → 그때 "필수값에서 빼 달라"고 요청하면, 코드에서 필수 검사만 조정하면 됩니다.

**정리:**  
- "INN·여권 필수로 둘래, 선택으로 둘래?"만 결정해 주면 됨.

---

## 4. 한 줄 요약 (초등학생용)

- **당신이 꼭 할 일:**  
  1) Supabase SQL Editor에서 **두 개 SQL 파일** (`SUPABASE_ORDERS_SHIPPING_COLUMNS.sql`, `SUPABASE_CS_DEFENSE_LOGS.sql`) 실행.  
  2) 관리자로 쓸 계정을 Supabase `profiles` 테이블에서 **is_admin = true**로 설정.

- **당신이 정하면 되는 것:**  
  - PG(결제사) 언제 붙일지, INN·여권을 필수로 둘지 선택으로 둘지.  
  - (Dadata는 이미 키 반영되어 쓰고 있으면 할 일 없음.)

- **이미 되어 있는 것:**  
  - 리뷰·주문은 목업 없이 실제 DB만 사용.  
  - 주문 시 배송 정보·금액 스냅샷 저장, 수령인 수정(관리자/고객), CS용 로그·결제 증거, PG 연동 대비 구조, 필수값 검사, 결제/DB 불일치 시 안내까지 모두 반영된 상태.

## 5. 가짜(테스트) 주문 구분·삭제

- **가짜 주문 넣는 방법**  
  결제 화면을 **`/checkout?test=1`** 로 열고 주문하면, 해당 주문만 `is_test = true` 로 저장됩니다. (일반 결제는 그대로 `is_test = false`)

- **구분**  
  - 관리자 → 주문 탭에서 **구분** 열에 «테스트» 배지로 표시됩니다.  
  - Supabase Table Editor / SQL에서 `WHERE is_test = true` 로 필터하면 테스트 주문만 볼 수 있습니다.

- **가짜 주문만 삭제 (중요 데이터는 그대로)**  
  Supabase 대시보드 → **SQL Editor**에서 아래만 실행하면 됩니다.  
  `is_test = false` 인 실제 주문은 전혀 건드리지 않습니다.

```sql
DELETE FROM public.orders WHERE is_test = true;
```

- **DB에 `is_test` 칸 넣기**  
  `docs/SUPABASE_ORDERS_SHIPPING_COLUMNS.sql` 를 한 번 실행해 두면 `is_test` 컬럼이 추가됩니다. (같은 파일에 배송 컬럼과 함께 들어 있음)

## 6. "주문 실패"가 났을 때 (실제 결제창이었어도 같은 실패인가?)

- **흐름:**  
  결제 버튼을 누르면 **① 주문 한 건을 DB(orders)에 먼저 저장**하고, 그 다음 **② 결제 승인(PG)** 을 부릅니다.  
  지금 오류는 **① 주문 저장 단계**에서 나는 겁니다. ② 결제창(PG)은 아직 호출도 안 한 상태예요.

- **그래서 실제 결제창이었어도?**  
  **네. 같은 시점에서 실패했을 겁니다.**  
  실제 PG를 붙여도, "주문 저장"이 실패하면 그 다음 단계인 결제 API는 호출되지 않고, 같은 "Не удалось оформить заказ" 메시지가 뜹니다.

- **왜 실패하는지 확인하는 방법**  
  - 주문 실패 시 알림에 **괄호로 (오류: …)** 가 붙어 있으면 그게 Supabase가 준 이유입니다.  
  - 또는 **F12 → Console** 에서 `[Checkout] order insert 실패 — 원인:` 뒤에 나오는 메시지를 보면 됩니다.  
  - 예: `column "receiver_name" of relation "orders" does not exist` → 배송 컬럼 추가 SQL을 안 돌린 경우.  
  - 예: `new row violates row-level security policy` → 본인 주문만 넣을 수 있는 RLS 때문에 막힌 경우(권한/정책 확인 필요).

- **가장 흔한 해결:**  
  Supabase **SQL Editor**에서 **`docs/SUPABASE_ORDERS_SHIPPING_COLUMNS.sql`** 전체를 한 번 실행해 주세요.  
  `receiver_name`, `receiver_phone`, `shipping_address`(와 필요 시 `items`, `is_test`)가 없으면 지금 코드가 넣는 값 때문에 insert가 실패합니다.
